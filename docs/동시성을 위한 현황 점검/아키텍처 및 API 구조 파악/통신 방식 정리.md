# 통신 방식

애플리케이션이 외부 클라이언트와 데이터를 주고받는 방식을 작성한 문서.  
요청이 어떻게 들어오고, 응답이 어떻게 나가느냐를 중심으로 작성되었다.

- 이 시스템에 어떻게 요청해야 하는지
- 응답이 어떤 형태로 오는지
- 인증이 어떻게 동작하는지

---

## 기본 설정

| 항목 | 값 |
|------|-----|
| 프로토콜 | HTTP REST |
| 포맷 | JSON (application/json) |
| 서버 포트 | 8080 |

---

## 공통 응답 구조

모든 API 응답은 `ApiResponse<T>` 포맷으로 반환된다.

```json
{ "code": "SUCCESS", "data": { ... } }
{ "code": "D001",    "data": null }
```

| 필드 | 설명 |
|------|------|
| `code` | 성공 시 `"SUCCESS"`, 실패 시 에러 코드 문자열 |
| `data` | 응답 데이터. 데이터가 없는 응답은 `null` |

HTTP 상태 코드는 헤더에서 확인하고, 비즈니스 결과 구분은 `code` 필드로 한다.

### 목록 조회 응답 — `PageResponse<T>`

목록 조회 API는 `data` 필드 안에 `PageResponse<T>` 구조로 응답한다.

```json
{
  "code": "SUCCESS",
  "data": {
    "content": [...],
    "totalCount": 100,
    "totalPage": 10,
    "page": 0,
    "size": 10
  }
}
```

| 필드 | 설명 |
|------|------|
| `content` | 조회된 항목 목록 |
| `totalCount` | 전체 항목 수 |
| `totalPage` | 전체 페이지 수 (`ceil(totalCount / size)`) |
| `page` | 현재 페이지 번호 |
| `size` | 페이지당 항목 수 |

## 에러 응답

예외가 발생하면 `GlobalExceptionHandler`가 모든 예외를 `ApiResponse` 포맷으로 변환한다.

### 에러 코드 체계

| 접두사 | 분류 | 예시 |
|--------|------|------|
| `D-XXX` | 도메인 비즈니스 에러 | `D001` 리소스 없음, `D005` 게시글 접근 권한 없음 |
| `A-XXX` | 애플리케이션 에러 | `A001` 인증 실패 |
| `C-XXX` | 공통(프레젠테이션) 에러 | `C001` 검증 실패, `C006` Body 파싱 실패 |
| `S-XXX` | 시스템 에러 | `S001` 서버 내부 오류 |

### 주요 HTTP 상태 코드 → 에러 코드 매핑

| HTTP 상태 | 상황 | 에러 코드 |
|-----------|------|-----------|
| 400 | 비즈니스 규칙 위반 | `D-XXX` |
| 400 | `@Valid` 검증 실패 | `C001` |
| 400 | Body 파싱 실패 | `C006` |
| 404 | 존재하지 않는 경로 | `C005` |
| 405 | 허용되지 않는 HTTP 메서드 | `C002` |
| 415 | 지원하지 않는 Content-Type | `C003` |
| 500 | 예상치 못한 서버 오류 | `S001` |

## 인증 방식 — JWT Bearer

### 토큰 스펙

| 항목 | 값 |
|------|-----|
| 헤더 | `Authorization: Bearer <accessToken>` |
| accessToken 만료 | 30분 |
| refreshToken 만료 | 7일 |

### 필터 적용 대상

`JwtAuthFilter`는 아래 경로에만 적용된다.

```
/posts, /posts/*, /users, /users/*
```

### 인증 흐름

```
HTTP 요청
  → JwtAuthFilter
      Authorization 헤더에서 토큰 추출
      → 유효: request attribute에 username 저장
      → 무효/없음: attribute 저장 없이 필터 체인 계속 진행
  → Controller
      @CurrentUser로 username 주입받음
      (인증 필요 API에서 username이 없으면 서비스에서 A001 반환)
```

인증 실패 시 요청을 즉시 차단하지 않는다. 인가 판단은 컨트롤러·서비스 레이어에서 수행한다.

---

## 요청 처리 흐름

```
HTTP 요청
  → JwtAuthFilter         (토큰 검증 → username을 request attribute에 저장)
  → Controller            (Request DTO → Command/Query 변환)
  → Service               (@Transactional 경계, 비즈니스 로직)
  → Repository → DB
  → Controller            (Result → Response DTO → ApiResponse 래핑)
  → HTTP 응답
```

- Request DTO는 Controller에서 Command/Query로 변환되며 도메인 레이어로 넘어가지 않는다.
- 예외가 발생하면 어느 레이어에서든 `GlobalExceptionHandler`가 받아 `ApiResponse` 포맷으로 변환한다.
