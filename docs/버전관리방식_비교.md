# 의존성/버전 관리 방식 3가지 비교

---

## 방식 1 — `build.gradle.kts`에 직접 버전 하드코딩
> 현재 우리 `oop` 프로젝트 방식

```kotlin
// build.gradle.kts (루트)
plugins {
    kotlin("jvm") version "2.2.21" apply false
    id("org.springframework.boot") version "4.0.4" apply false
}
```

| | |
|--|--|
| ✅ 장점 | 파일 하나만 보면 됨. 설정 단순. 작은 프로젝트엔 충분 |
| ❌ 단점 | 버전이 여러 모듈에 흩어지면 관리 어려움. 오타 위험. IDE 자동완성 없음 |

---

## 방식 2 — `gradle.properties`로 버전 분리
> 중간 단계

```properties
# gradle.properties
kotlinVersion=2.2.21
springBootVersion=4.0.4
group=com.oop
version=0.0.1-SNAPSHOT
```
```kotlin
// build.gradle.kts
val kotlinVersion: String by project
plugins {
    kotlin("jvm") version kotlinVersion apply false
}
```

| | |
|--|--|
| ✅ 장점 | 버전을 한 파일에서 관리 가능. 설정값과 빌드 로직 분리 |
| ❌ 단점 | 단순 문자열이라 IDE 자동완성 없음. 라이브러리 정보(group, name)는 여전히 build.gradle에 흩어짐 |

---

## 방식 3 — `libs.versions.toml` (Version Catalog)
> `hexagonal-module-sample` 방식. Gradle 공식 권장 방식 (Gradle 7.4+)

```toml
# gradle/libs.versions.toml
[versions]
kotlin = "2.2.21"
spring-boot = "4.0.4"

[plugins]
kotlin-jvm     = { id = "org.jetbrains.kotlin.jvm",    version.ref = "kotlin" }
spring-boot    = { id = "org.springframework.boot",    version.ref = "spring-boot" }

[libraries]
spring-web = { group = "org.springframework.boot", name = "spring-boot-starter-web", version.ref = "spring-boot" }
```
```kotlin
// build.gradle.kts
plugins {
    alias(libs.plugins.kotlin.jvm) apply false
    alias(libs.plugins.spring.boot) apply false
}
dependencies {
    implementation(libs.spring.web)  // IDE 자동완성 + 타입 안전
}
```

| | |
|--|--|
| ✅ 장점 | 버전/라이브러리 **중앙 집중 관리**. **IDE 자동완성** 지원. **타입 안전**. 여러 모듈에서 동일 참조 보장. Gradle 공식 표준 |
| ❌ 단점 | 초기 설정 비용 있음. toml 파일 문법 추가 학습 필요. 작은 프로젝트엔 오버엔지니어링일 수 있음 |

---

## 한눈에 비교

| | 방식 1 (하드코딩) | 방식 2 (gradle.properties) | 방식 3 (Version Catalog) |
|--|--|--|--|
| **버전 중앙 관리** | ❌ | ✅ (버전만) | ✅ (버전 + 라이브러리 전체) |
| **IDE 자동완성** | ❌ | ❌ | ✅ |
| **타입 안전** | ❌ | ❌ | ✅ |
| **설정 복잡도** | 낮음 | 낮음 | 중간 |
| **Gradle 공식 권장** | ❌ | ❌ | ✅ |
| **적합한 프로젝트 규모** | 소규모 | 소~중규모 | 중~대규모 |

---

## 추천

### 지금 oop 프로젝트 → **방식 1 유지 (or 방식 3으로 업그레이드)**

> **현재 학습 목적의 소규모 멀티모듈 프로젝트라면 방식 1로 충분합니다.**  
> 모듈이 4개 정도이고 의존성도 많지 않아서 관리 비용이 크지 않음.

단, **실무나 팀 프로젝트를 목표로 한다면 방식 3(Version Catalog)을 지금 익혀두는 걸 추천합니다.**  
이유:
- Gradle 7.4부터 공식 표준, Spring Initializr도 기본으로 생성해줌
- IntelliJ에서 toml 자동완성 완벽 지원
- 의존성이 20개, 30개 넘어갈 때 방식 1·2는 관리가 급격히 어려워짐
- `hexagonal-module-sample` 같은 실무 프로젝트에서 실제로 사용하는 방식
